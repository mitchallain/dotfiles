# Agent Tasks

Manage projects and tasks in the agent notes knowledge graph.

**Notes root:** `/home/mallain/notes/12-agent-notes/`

## Directory Structure

```
{notes_root}/
├── 00-projects/
│   ├── _registry.md                          # Project index (regenerated by /agent-tidy)
│   ├── {NN}-{domain-name}/
│   │   └── {NN}-{area-name}/
│   │       ├── .counter                      # Next project number (bare integer)
│   │       └── {NN}-{project-name}/
│   │           ├── {project-name}.md         # Main project note (unique filename)
│   │           ├── .counter                  # Next task number (bare integer)
│   │           └── {research-topic}.md       # Research notes (unique filenames)
├── 01-notes/                                 # Knowledge notes (managed by /agent-notes)
├── 02-archive/                               # Completed/abandoned projects
│   └── (mirrors 00-projects/ structure)
└── log.md                                    # Append-only activity log (shared)
```

Directories are named `{NN}-{kebab-case-name}` with NN zero-padded to 2 digits,
numbered sequentially within their parent. Counters initialize at `0`.

## ID Scheme

Human-readable IDs for quick reference in conversation and task lists.

| Level   | Format                           | Example        |
|---------|----------------------------------|----------------|
| Domain  | `{ACRONYM}`                      | `HL`            |
| Area    | `{DOM}-{ACRONYM}`                | `HL-API`        |
| Project | `{DOM}-{AREA}-{NNN}`             | `HL-API-003`    |
| Task    | `{DOM}-{AREA}-{PROJ}-{TASK}`     | `HL-API-003-001`|

- Domain and area acronyms: 2-5 uppercase letters, chosen at creation time
- Project and task numbers: zero-padded to 3 digits
- IDs are immutable — never change once assigned, even after rename or archive

The directory number (`NN`) and the project number in the ID (`NNN`) are assigned
from the same `.counter`, so they stay in sync. Directory uses 2-digit padding,
ID uses 3-digit: directory `05-foo/` has ID `...-005`.

## Note Format

### Project note

The main project file lives inside the project directory and has a unique filename
matching the directory's kebab-case name (without the number prefix).

```markdown
---
id: HL-API-003
title: Add Rate Limiting to REST API
type: project
status: active
priority: P1
created: 2026-02-15
updated: 2026-03-09
tags: [api, rate-limiting, redis]
domain: HL
area: API
---
# Add Rate Limiting to REST API

Related:: [[redis-sliding-window]], [[api-auth-middleware]]

## Plan

Token bucket algorithm backed by Redis. Configurable per-endpoint limits
via config file. See [[sliding-window-vs-fixed-window]] for algorithm research.

## Tasks

- [x] HL-API-003-001 Design rate limit configuration schema
- [ ] HL-API-003-002 Implement Redis-backed token bucket
- [ ] HL-API-003-003 Add integration tests
```

### Research note

Research notes live alongside the project file. Each has a unique filename.

tags: [rate-limiting, algorithms, redis]

### Frontmatter fields

**All notes:** `title`, `type`, `created` (required)

**Projects additionally:** `id`, `status`, `priority`, `domain`, `area`, `updated`

**Research additionally:** `project` (wikilink to parent)

### Status values

- `active` — being worked on or planned
- `blocked` — waiting on external dependency
- `complete` — all tasks done
- `abandoned` — no longer pursued

### Priority values

- `P0` — urgent, blocking others
- `P1` — important, do soon
- `P2` — planned, when time allows
- `P3` — backlog/someday

## Wikilinks

Use `[[filename]]` (without `.md` extension) to link between notes. Filenames must
be globally unique across the vault so Obsidian resolves them unambiguously.

When creating or modifying any note, search for related notes and add
`[[wikilinks]]` to them. Also add backlinks from those notes to the new one.

## Procedures

### Create a domain

1. List `00-projects/` to find the next available number.
2. Create `00-projects/{NN}-{domain-name}/`.
3. Choose an acronym. Record it in a comment at the top of the directory
   (or just in the project notes that use it — the acronym is self-documenting).

### Create an area

1. List the domain directory for the next available number.
2. Create `{domain}/{NN}-{area-name}/`.
3. Create `.counter` with contents `0`.
4. Choose an acronym.

### Create a project

1. Read the area's `.counter` to get the next project number.
2. Create `{area}/{NN}-{project-name}/` (NN = counter value, zero-padded to 2 digits).
3. Create `{project-name}.md` inside using the project note template.
   - The `id` field uses the counter value zero-padded to 3 digits.
4. Create `.counter` in the project directory with contents `0`.
5. Increment the area's `.counter` and write it back.
6. Search for related notes and add `[[wikilinks]]` in both directions.
7. **Log** — append to `log.md`: `## [{date}] create-project | {ID} {Title}`

### Add a task

1. Read the project's `.counter` for the next task number.
2. Append to the Tasks section: `- [ ] {ID}-{TASK} Description`
   (task number zero-padded to 3 digits).
3. Increment the project `.counter` and write it back.
4. Update the `updated` frontmatter field.
5. **Log** — append to `log.md`: `## [{date}] add-task | {TASK-ID} Description`

### Complete a task

1. Change `- [ ]` to `- [x]` for the task line (and any subtasks).
2. Update the `updated` frontmatter field.
3. **Log** — append to `log.md`: `## [{date}] complete-task | {TASK-ID} Description`
4. If all tasks are now complete, ask the user if the project should be archived.

### Archive a project

1. Set frontmatter `status: complete` (or `abandoned`).
2. Move the project directory to the mirror path under `02-archive/`.
   Create intermediate directories in `02-archive/` as needed.
3. **Log** — append to `log.md`: `## [{date}] archive | {ID} {Title} ({status})`

### Add a research note

1. Create `{descriptive-name}.md` in the project directory.
2. Use the research note template with `project: "[[project-name]]"`.
3. Add a `[[wikilink]]` to the research note from the project's Plan section.

## Activity Log

`{notes_root}/log.md` is an append-only chronological record of all wiki activity.
Shared across all agent skills. See `/agent-notes` for full format specification.

Each entry: `## [{date}] {operation} | {subject}`. Append to the end of the file.
Create `log.md` if it doesn't exist, starting with `# Activity Log\n`.

## Permissions

- **Read:** any file under `/home/mallain/notes/`
- **Write:** only within the notes root

## Rules

1. **Filenames are globally unique.** No two `.md` files in the vault share a name.
2. **Counters always increment.** Never reuse a number. Increment immediately after use.
3. **IDs are immutable.** Never change an assigned ID.
4. **Projects are finite.** Split if scope grows unbounded.
5. **Link generously.** Add `[[wikilinks]]` to all related notes when creating or editing.
